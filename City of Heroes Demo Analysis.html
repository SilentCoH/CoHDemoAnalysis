City of Heroes Arena PvP Demo Analysis¶

Code and Tutorial created by @Silent Method, July 2020

Introduction¶

In order to pull some basic stats from 8v8 Arena matches (and save myself a ton of time trying to do the same in VOD reviews... and be a huge nerd), in early July 2020 I wrote code in R to parse the output of .cohdemo files. The code posted here can be run in R, which is downloadable for free at https://cran.r-project.org/bin/windows/base/ (Windows link), though you may also want RStudio (https://rstudio.com/products/rstudio/download/) if you plan to do any editing.

Why use this?¶

City of Heroes offers virtually no Arena statistics other than kill counts (which tend to mean very little, especially from a single match). Parsing chat logs can be done by individual players to see their exact damage output - fun note, you can use damage badge progress for a shortcut - but that's time-consuming, individualized, and raw damage output isn't all that important by itself either.

This code uses some built-in assumptions to try to identify spikes in a typical Arena match and figure out who used which powers, at what time, including both the offensive and defensive (evasion/healing) side of things. This can be useful in determining areas to improve as well as figuring out why a match occurred the way that it did.

In the past few weeks of testing and using this code, Renegades players have used it on a voluntary basis for their own knowledge and improvement. Now, in releasing it more broadly, people are free to do with it as they please. With that said, in a small PvP community that exists only as long as we are able to find people willing to play against each other, using match statistics to beat people over the head - teammates or opponents - probably isn't the best use of anyone's time. Everyone I've played with / against has had good and bad matches, both offensively and defensively, so keep that in mind as well.

Many of the metrics provided by this code start to make more sense after reviewing not one, but several matches - good ones, bad ones - and getting a sense for how they look when things work out well vs. when they don't.

Finally, users may edit this code and improve on it, or they may develop their own, as a few others started working on in July. Use whatever works best for you. My goal in providing this publicly is to give a user-friendly tutorial for one option that has worked well over the last several weeks.

Why would I need to edit the code?¶

Mostly because it's incomplete. I've built it to handle situations I've seen while recording first person demos (i.e. as one of the participants in the 8v8 match) over the past few weeks, but during that time I:

  • Never ran into someone using the Dark Blast set
  • Never ran into someone using Pain instead of Empathy
  • Did not encounter many of the weird entity names that could screw up entity detection
  • Only rarely attempted to use this for anything less than 8v8s (I have parameters to make this work, but it's only been lightly tested)
  • Only once attempted to use this for Spectator mode demos (though it should work if whoever is running the code fills in entities properly)
  • etc.

...so you could easily need to add to it. Alternatively, if this is hosted on github, other users or I could periodically add these features when they arise and update the code.

What if I have a question about the meaning of the output, or I run into an issue I can't fix?¶

Message me on the Homecoming PvP Discord: @Silent Method

Other players in the community may make improvements to the code over time, or I may periodically release updated versions with some of my own fixes.

How do I use it?¶

  • Download the two .R code files if you haven't done so already
  • Load into an Arena match
  • Type "/demorecord DEMONAME" into chat (a lot of times I will use d1, d2, d3 for the first matches of the night and then rename them later)
  • Play (or spectate) the match
  • Optionally, at the buzzer at the end of the match, type or hit a macro for /demostop . Otherwise the demo will stop when you leave the zone, but it may pick up kills that occur after the buzzer
  • Open R or RStudio
  • If you are in R, you will probably want to open a new script (File -> New Script) so that you can copy some of the code below and then run highlighted portions (highlight -> Ctrl+Enter), where the output will appear in the console (stretch the console window as needed). If you are in RStudio, which requires you to also have R installed, then you should already have a script and console window.
  • If you viewed the match as a spectator, then you will want to set checkEntities = TRUE and modify the team assignments before proceeding. You may also need to temporarily modify the exclNames parameter to remove your own character as an entity. This code was written primarily with first-person demos in mind, recorded by a match participant rather than a spectator.
  • Run a few bits of code in R / RStudio (minimalist example below) and inspect the output:

What if I want to kick the output out to Excel?¶

That can easily be done via write.csv (or export from the rio package, which would need to be installed, etc.). Any table of output can be written to a .csv file, possibly modifying it first. An example is below, though I have tended to use this code more for review of specific matches rather than for aggregating data across matches, as team comps, attack chains, etc. tend to shift over time.

## Example output to .csv (not run)
## The file will be output to your current working directory - getwd().
## Users can easily write a loop to throw different outputs into different filenames, following the example here:
demoname <- "072220_4"
out1 <- parseDemo(demoname, checkEntities = TRUE)

teamIndex <- 0
write.csv(analyzeGroupOffense(out1,teamIndex), paste0("offenseData_", demoname, "_team_", teamIndex, ".csv"),
         row.names=FALSE, na="")
teamIndex <- 1
write.csv(analyzeGroupOffense(out1,teamIndex), paste0("offenseData_", demoname, "_team_", teamIndex, ".csv"),
         row.names=FALSE, na="")
## ...etc. Users may instead want to output an entire scrim night's worth of data; see the bottom of the Example below
##    for code to rbind multiple matches' results.

A quick note on Entities - the most likely obstacle to the code running smoothly¶

If you run the code exactly as shown above, you'll be prompted to check entities. Demos identify entities as players, pets, and so forth. Here is an example from a recent match, and you can see that it makes its best guess about team membership, which should usually work for first-person demos but requires some modification for spectator demos (more on that in a bit):

Follow the prompts to either confirm the list or enter editing mode, where you can declare each entity to be a player (or not) and assign them to a team. If you don't have exactly expectedPlayers number of players after this point, the code will error out. expectedPlayers is a parameter - covered later on - indicating how many players to expect in the match. By default, it is 16, which covers 8v8s. You should be able to modify it without issues, but it has not been tested extensively outside of 8v8s.

The spike comparison contains a basic summary showing:¶

  • Approximate score of the match (not perfect but probably correct 95% of the time; once in a while, deaths get missed, and kills after the buzzer will count unless you use the customEnd parameter. Using /demostop at the buzzer will improve accuracy)
  • Approximate number of spikes fired (can tune spike detection with parameters)
  • Average number of players who fired at least one attack per spike
  • Average number of total attacks fired per spike
  • Average time (in seconds) between the first activated power on spike and the last one (does not count animation time), i.e. the duration of firing
  • Average "timing difference" (larger values = players' first attacks on spike were more spaced out, which is bad. This metric will be explained later)

Offensive summaries¶

  • targeted = approximate number of times targeted. This is incredibly important context for pretty much everything else in the output. For example, someone who gets targeted 20 times probably should not be on much more than 50-60% of spikes at the absolute maximum, unless their whole team is winning so handily that they can stay on target even while being targeted.
  • on_spike_pct = percentage of spikes where the player fired at least 1 attack. Does not distinguish between firing at someone who is affectable vs. phased vs. dead before the damage arrives.
  • attacks_per_spike = average attacks fired per spike that the player was on.
  • first_attack_timing = average time in seconds that the player was behind whoever fired first on the spike. Note that this will always be a positive value unless you fire first on every single spike that you're on, in which case it would be 0. Higher values = later to the spike.
  • first_attack_rank = same as above but instead it's your ranking (first attacker is 1, second attacker is 2, etc.). Higher values = later to the spike.
  • first_timing_distance = average difference in seconds between your first attack and everyone else's, across each spike. Negative values = tend to be later than your teammates; positive values = tend to be earlier than your teammates. Players who fire with the team will be close to 0, and you generally want the whole team to be closer to 0. Though keep in mind it's still averaged across all spikes. As you review more and more matches, you will start to get a feel for what seems like good vs. bad values for this metric.
  • chain_duration = average firing duration, correlated strongly with attacks_per_spike. This is the time in seconds between the first attack fired on the spike and the last attack fired. It does not do anything with animation times (but that could be added).
  • compactness = not sure how useful this is. It's chain_duration / attacks_per_spike, so lower values are better. You would have high values if you typically fired 1 attack, got an "out of range" warning, ran to catch up, fired another attack... etc. It's trying to measure how quickly users launch their attacks in succession, but it's not actually that useful IMO.

Note: One useful thing to keep in mind is who's target calling. Target callers tend to have "higher than expected" (relative to the number of times they were targeted) on_spike_pct and also tend to fire a bit earlier, on average.

Defensive summaries¶

This contains a few snippets of output:

  • powerUsage contains targeted, deaths, survival_rate, and amount of power usage for any powers defined as evasion powers.
  • defenseData contains some more individual data:
  • avg_players_on_spike = average number of attackers when this person was targeted.
  • avg_attack_count = average number of attacks fired when this person was targeted.
  • evaded = percentage of attempts where this person took some evasive action around the same time. This is counted regardless of whether the evasion was "successful."
  • healed = percentage of attempts where someone attempted a heal on this person (again, regardless of whether it was successful or not).
  • team_survival_rate, team_evade_rate, and team_heal_rate are simple averages across all spikes for how often players survived, evaded, and were healed. Note that better evasion rates likely have an inverse relationship with better heal rates - i.e. players/teams who evade more often are less likely to need (or receive) heals.

Healing summaries¶

Very similar idea to the offensive summaries:

  • targeted still shows up here. Again, it's very important context.
  • on_heal_pct = percentage of spikes against this team where the player attempted a healing power. Keep in mind the team-wide evasion rates; higher rates of evasion mean the healers will need to be on target less often.
  • heals_per_spike = average number of healing powers fired per spike.
  • first_heal_timing = average time in seconds after the spike began that the player activated a heal.

There is more work that could be done here. Admittedly, I haven't fleshed out this part too much. Note that it will catch "off-healers" too, e.g. Indecent using an Elec Affinity power on spike at one point, but this part of the code only captures powers used in the vicinity of a spike. If you want to see what players are using between spikes, see the Power Summary below.

Finally, sometimes it's useful to aggregate data from a particular night of scrimmages. This can be done using the usual R commands, though I haven't rolled them up into functions (mostly because this is very quick to run with base R functions anyway). Suppose you have out1, out2, out3, out4, out5 stored in memory as five matches. Then you could do this to pull together offensive summaries for Team 0:

In [ ]:
## Not run
offensiveTotal <- rbind(analyzeGroupOffense(out1,0),
                       analyzeGroupOffense(out2,0),
                       analyzeGroupOffense(out3,0),
                       analyzeGroupOffense(out4,0),
                       analyzeGroupOffense(out5,0))
offensiveTotal$match <- rep(1:5, each = 8) ## Fills in "match" variable as 1 through 5 for an 8v8

...then you could do more interesting stuff like filtering out the supports and ordering the stats by player or match.

For defensiveTotal, remember that the defensive group output has several components, so you'd have to do something like this:

In [ ]:
## Not run
defensiveTotal <- rbind(analyzeGroupDefense(out1,0)$defenseData,
                       analyzeGroupDefense(out2,0)$defenseData,
                       analyzeGroupDefense(out3,0)$defenseData,
                       analyzeGroupDefense(out4,0)$defenseData,
                       analyzeGroupDefense(out5,0)$defenseData)
defensiveTotal$match <- rep(1:5, each = 8) ## Fills in "match" variable as 1 through 5 for an 8v8

  • x : demo name in quotes, without the .cohdemo extension, e.g. "d1".
  • keySkills : a set of abilities to track. By default (i.e. when NULL), this will be comprised of the union of attackset, healset, evadeset, and otherset (below).
  • exclNames : a set of entity names to exclude as non-players. Basically, the demo assumes anyone with an entity ID under 100 will be a player (since players load first), but sometimes other entities get spawned under 100 as well. I continually add to this list if checkEntities discovers more non-player names.
  • attackset : a set of ability names to count as attacks (i.e. counted as an offensive part of spikes). These ability names are defined later in powerset. Note that spikes must be aimed at a particular player, so randomly throwing abilities at other players, off-spike, will not count as being on spike.
  • healset : a set of ability names to count as heals in heal tracking.
  • evadeset : a set of ability names to count as evasive maneuvers in evasion tracking.
  • otherset : a set of other ability names to track usage but not count as part of spikes / evasion / healing.
  • spikeWindow : time in seconds before a series of attacks aimed at a particular player is no longer counted as part of the same spike. Default 3.5. What this means is that if Target X goes 3.5 seconds without anyone on the other team firing an attack at them, then the spike is over. If even one player continues firing at Target X within 3.5 seconds of anyone's previous attack, then the spike continues.
  • min_attacks_per_spike : minimum number of attacks that must be fired at a player in order for it to count as a spike attempt. Default 2. There are exceptions to this (see below).
  • max_time_per_spike_sec : maximum time in seconds that a spike can last. Default 11. If someone keeps extending the spikeWindow by continuously hitting a target, then a new spike will begin after 11 seconds. In practice, this has not happened often, given that most teams are not currently roguing, but roguing could mess with this system. This is intended to catch "re-call" spikes as being separate, which it does fairly well.
  • hitSupportCredit : logical indicator for whether hitting a "support" during a spike window still counts as being "on spike" even if that support is not the target. Default TRUE. Why? Sometimes forcing evasion or mezzing a support during a spike can contribute to the spike's success. Currently, "supports" are automatically defined as the two players on each team who use the fewest powers from attackset. This is done deeper in the code but could be reworked to feed in the support names via parameters.
  • healWindow : in order to be counted as being in response to a spike, heals must be attempted between the spike's start time (first attack) and the spike's end time (when the last attack of the spike is fired), plus healWindow seconds. Default 2.
  • preEvadeWindow : a window of time that allows spikes to be counted (and specifically, counted as evaded) even if they do not meet the min_attacks_per_spike requirement. Default is c(-2, 1) which means that if a player takes an evasive action like Jaunt anywhere from 2 seconds before, to 1 second after, the first attack in a spike... the spike will be counted regardless of min_attacks_per_spike settings. A few notes: 1) there is probably no good way to count pre-evasions that result in 0 attacks fired, and 2) in order to count as an evasion, there's also a hard-coded maximum limit of 2 seconds after the last attack in the spike is fired (similar to healWindow above, but it's hard-coded, rather than a parameter). This will occasionally pick up an extra spike that isn't really a spike. However, there is no surefire way to give credit to evasions involving only 1-2 attacks without at least sometimes counting an extra spike, so it's up to the user. Consistency across matches is more important than worrying about which error to favor.
  • customEnd : time in seconds after which all demo information is ignored and the match is considered over. Default NULL, which means that the demo will either track data until a teammate (team 0) leaves the game after 10 minutes of recording, or until the demo ends if no teammate is seen leaving the match. However, if you know the match lasted for exactly 615 seconds of demo recording - e.g. because you started recording at exactly 0:15 until match begins - then you could supply 615 as the value.
  • checkEntities : logical indicator for whether users should be prompted to confirm the players / teams are correct. If they are not, then the user is given the option to identify players and teams directly. Default TRUE. Can be turned to FALSE for demos that have been successfully parsed before or where entity mismatches are unlikely.
  • expectedPlayers : number of players expected in the match. Default 16, corresponding to 8v8. If the number of tracked entities is not equal to this value after checking entities, then the function will return an error code.
  • emotePolice : logical indicator for whether to track emote usage by players. Default TRUE. This is an incredibly basic function that only returns what emotes were used; there are some built-in definitions for emote canceling like calling out usage of /em liedown or /em shocked, as well as one that calls out /em holdtorch (a non-canceling emote) as probably just being me messing around at the start of the match.
  • textset and powerset : sets of power activation effect names and corresponding power names (which are then used by attackset, healset, etc.). These are 1-to-1, and ordering matters, so if you add to textset, you should add to powerset in the same order. You will need to add to these sets if there are any new powers I haven't already captured, like the examples of Dark Blast and Pain that I gave earlier. Note that because I am allowing "any" matches to the text rather than exact matches, it is technically safest to start with the / in the FX name like /ARROW_NET.FX because otherwise there can be duplicate matches. POISONLIQUIDPROJECTILE.FX would actually cover both Weaken and Envenom, which are PALMPOISONLIQUIDPROJECTILE.FX and POISONLIQUIDPROJECTILE.FX respectively, so I use /POISONLIQUIDPROJECTILE.FX for Envenom to differentiate it.

The most likely parameters a casual user would need to modify are:

  • exclNames to add to the list of entities to exclude.
  • textset and powerset to add powers, and then add them to tracking them by adding to attackset, healset, evadeset, otherset. Remember that attackset is used for finding spikes, so I tend not to include off-target harassment powers like Siphon Speed and Weaken as "attacks." I put those powers in otherset instead.
  • expectedPlayers if using this for something other than 8v8s.

Details from parseDemo¶

  • A general note about some of the arbitrary decisions / default parameter settings that are noted both above and below: many of these have proven highly consistent with actual, detailed VOD reviews. You're free to modify them if you like, but the default settings (including some hard-coded decisions) are fairly accurate most of the time.

Technical Notes, Settings, and Known Issues¶

  • I detect player deaths in two different ways, as seen in the code, and then try to de-duplicate between them. I do this because sometimes one method catches it while the other doesn't.
  • Player entities assume that players will have entity numbers < 100 (generally true since they load first).
  • Default team assignment assumes that the first 8 (or expectedPlayers/2) players seen are on Team 0, and the second 8 seen are on Team 1. This is true for first-person demos but not necessarily true for spectator demos.
  • Spirit Shark Jaws has no .FX, but it has a MOV associated with it, so I created my own fake "FX" coding based on that usage. This means SSJ does not have a target (unless I use "hit" animations - more on that in a moment).
  • Strangler is basically a ghost, because it has no .FX and no MOV. As a result, Strangler is only detected when it causes a "hit" animation from the target (which still gives information about who fired it and who the target was). So its detection is different from all other powers.
  • Blind had to be dealt with in a special way because the targets of the Blind also "cast" Blind on themselves. So basically I remove opposing team instances of Blind if they occur right after an attacker uses Blind. This was implemented before I started using targeting data, but it works extremely well.
  • Phase Shift sometimes appears in quick succession, probably when people tap it on but then get mezzed and then re-cast it later. I wrote code to only count the last instance within a quick succession.
  • Spikes require a target, as noted above. However, projectile attacks that miss (e.g. failing a 95% roll) will not have a target. So I wrote code to assign any missed attacks to the nearest possible spike. This isn't perfectly accurate, but without any valid target, there's no way to be sure.
  • Speaking of which, I was going to pull hit/miss rates for players, but then I realized (per the notes above) that it would only work for projectile attacks, so I decided not to bother. There is some commented out code which could be repurposed to do this, but the denominator would have to be only projectile attacks.
  • A few powers have identical FX. Fortitude and AB do, so they have to be grouped together. The biggest example is Charged Shot and Penetrating Ray... unfortunately there is no good way to distinguish between them, so they are also grouped. Other powers are grouped together despite different FX, like Raptor Pack, Fly, and Mystic Flight, because they serve the same purpose.
  • I trimmed false "spikes" where 1 stray attack hits somebody and another stray attack hits a support at the same time (when hitSupportCredit is TRUE). Those should not be counted as spikes. This has worked well in practice based on VODs.
  • I also hard-coded the removal of "accidental" follow-ups to a spike, e.g. casting Blazing Bolt at someone by accident because the previous target died quickly. The criteria: If <= 3 attacks are fired at someone <= 2 sec from the firing of the last attack from the previous spike, then it's probably an accident.
  • Recall spikes (i.e. hitting the same target again) are handled a little imperfectly by max_time_per_spike_sec. Usually, clean recall spikes will still have at least 3.5 seconds (default spikeWindow value), so they'll be counted separately anyway, as they should be. However, there are sometimes dirty recall spikes where people keep firing, and in that case the separations between spikes are based on max_time_per_spike_sec (default 11). But my code is a little lazy, and I basically just assume there won't be more than 4 dirty recall spikes in a row.
  • Spike success rates are not 100% synonymous with kill/death counts. Why? Because a spike is only linked to a success if the target dies between spike end (i.e. the last attack being fired) plus 4 seconds, hard-coded. In weird cases like phase chasing or roguing that occurs after a spike, I didn't want to drag the whole spike out that long, so I call the success marker (in spikeDetails) "approx_success" to indicate that it's not 100% in sync with actual kill and death counts.

FAQ¶

  • Why don't you use "hit" animations more often than just Strangler? - the goal is to find out who's casting abilities and when they're doing it. You could get useful information out of "hit" animations, such as when the attacks land, when the heals land, and so forth... but lots of things can interfere with hit animations. For instance, Hibernated targets don't show "hit" animations. If someone Hibernates almost instantly, that's good play by the evading player, but the offense shouldn't be penalized as if they didn't cast any attacks. Or alternatively, if a really clean spike comes in and the target explodes, the person who was quick to try to heal them shouldn't be penalized as if they didn't cast a timely heal; it was just good play by the offense. So instead I chose to steer away from "hit" animations as much as possible and only use them when absolutely required to, i.e. Strangler. Even for Spirit Shark Jaws, I prefer to use the animation (from its associated MOV) rather than the "hit" animations. Or to phrase it simply... analysis based on "hit" animations would have a lot more caveats and issues than analysis based on casting animations, so I chose the casting route whenever possible.
  • Why aren't damage values available? - damage values in the demo appear as floatdmg, but floatdmg appears very inconsistently, and it's mostly based on whether you're close enough to the player that you can see numbers overhead. You could arbitrarily code each attack with an average damage value, but that's inaccurate in terms of actual damage dealt anyway because some cast attacks will miss or hit a phased/dead target.
  • Why are durations based on first attack fired to last attack fired, instead of actual power animation times? - because I've been too lazy to add animation times along with textset and powerset, and then incorporate it into summary code. It wouldn't add all that much to the analysis since it will still correlate strongly with the current spike duration metric (time the first power was cast until time the last power was cast) anyway.

Other Analysis Function Details¶

The other analysis functions are intended to be very simple. They all operate on the output produced by parseDemo, but they tidy up summaries in a useful fashion, either team-wise ("Group") or individual ("Player"). I won't cover these in much detail, but here are the functions and their parameters:

  • analyzePlayerOffense(parseOut, name) : e.g. analyzePlayerOffense(out1, "Doom Reverb"). Returns a list with several objects, including the player's contribution to each team spike, several metrics, and a chain_table showing what attack chains they used.
  • analyzeGroupOffense(parseOut, teamNum) : e.g. analyzeGroupOffense(out1, 0). Pulls some of the metrics from analyzePlayerOffense into tables for the entire team (0 or 1).
  • analyzeCustomGroupOffense(parseOut, name1, name2, name3 ... ) : similar to analyzeGroupOffense, but instead of a team, it lets you feed in up to 8 names (must supply at least name1 and name2), which could be on opposite teams, to get summaries.
  • analyzePlayerHealing , analyzeGroupHealing , analyzeCustomGroupHealing : see above, same idea.
  • analyzeGroupDefense(parseOut, teamNum, evadeset = c("Phase Shift","Hibernate","Jaunt","Raptor or Fly")) : defensive information (deaths, targets, evasive powers used) is more compact and straightforward than offensive or healing data, so I only made a group summary function that returns several outputs in a list: powerUsage shows uses of the evadeset powers, defenseData shows some detailed information, and then team_survival_rate, team_evade_rate, and team_heal_rate are returned (as explained in notes up above).

Wishlist¶

It would be pretty easy to use R's base graphics, or the ggplot2 package (install.packages("ggplot2") to install the first time only, then library(ggplot2) to read it into memory each time you load R), to create line graphs of spikes over time, kills over time, rolling success rate over time, number of players on spike over time, etc. I haven't done any of that yet, but it would be pretty easy if you pick apart the spikeSummary and spikeDetails portions of the output.

There are plenty of other cool analytical things that could be done or added, though I will say that I tend to prefer the metrics this code provides - average times, average counts - over black and white classification attempts, e.g. "good spike vs. bad spike." Setting aside the fact that that classification depends not just on firing but also on how the damage lands (including the debuff), it's also the age-old classification vs. regression debate. Getting averages across a match's spikes, or a team's set of scrimmages, and getting a feel for the difference between those is often more useful than trying to fit everything into a good or bad bin. But any of those approaches are possible with small modifications to this code.
